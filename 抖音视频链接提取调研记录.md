# 抖音视频链接提取调研记录

本文记录一次基于 Chrome CDP 和 `playwright-core` 提取抖音视频真实播放/下载地址的调研过程。目标不是做一个完整下载器，而是先验证一条抖音分享短链在远程浏览器环境里，是否可以稳定拿到视频流地址，并进一步触发浏览器下载。

## 调研目标

本次调研围绕三个问题展开：

1. 能否通过远程 Chrome CDP 打开抖音短链并落到目标视频页。
2. 能否从页面请求里提取出真实的视频播放地址和下载地址。
3. 能否在远程浏览器所在机器内，模拟一次“浏览器自己发起下载”的动作。

## 基本环境

- 远程 Chrome 通过 CDP 暴露在 `192.168.31.112:9223`
- 使用 `playwright-core` 的 `chromium.connectOverCDP()` 连接远程浏览器
- 脚本运行在本地工作环境，但实际页面行为发生在远程浏览器中

关键认识：

- `curl` 在哪台机器执行，文件就会下载到哪台机器。
- `connectOverCDP()` 控制的是远程浏览器，因此如果在页面里触发下载，文件会落到远程浏览器所在机器。

## 方案演进

### 第一阶段：验证短链可以打开

先验证抖音短链是否能通过 CDP 正常打开，并确认页面可以落到真实视频页。

这一步的核心不是提取数据，而是确认：

- CDP 连接正常
- `page.goto()` 能完成跳转
- 页面能加载到抖音视频页

### 第二阶段：提取视频链接

最开始尝试过几种思路：

- 从页面 DOM 里的 `video.currentSrc` 提取
- 从性能资源列表里筛视频资源
- 从 CDP 网络事件里筛 `video/*`
- 从抖音详情接口中直接提取

最终证明最稳定、最清晰的路径是：

- 监听页面的网络响应
- 抓取 `aweme/v1/web/aweme/detail`
- 从接口 JSON 中读取 `video.play_addr.url_list` 和 `video.download_addr.url_list`

也就是说，真正的核心不是猜测页面里哪个请求是视频，而是直接拿抖音视频详情接口返回的数据。

### 第三阶段：极简脚本

在确认详情接口路径有效之后，把脚本收敛成最小版本，只保留：

- 连接 CDP
- 打开页面
- 监听 `aweme/detail`
- 提取 `playAddr` 和 `downloadAddr`

对应的核心代码如下：

```js
const { chromium } = require("playwright-core");

const CDP_WS = process.env.CDPX_WS;
const TARGET_URL = process.env.TARGET_URL;

async function main() {
  let result = null;

  const browser = await chromium.connectOverCDP(CDP_WS);
  const context = browser.contexts()[0] || (await browser.newContext());
  const page = await context.newPage();

  page.on("response", async (response) => {
    const url = response.url();
    if (!/\/aweme\/v1\/web\/aweme\/detail\//.test(url)) return;

    const json = await response.json();
    const detail = json?.aweme_detail;
    const video = detail?.video;
    if (!video) return;

    result = {
      awemeId: detail.aweme_id || "",
      desc: detail.desc || "",
      playAddr: video.play_addr?.url_list || [],
      downloadAddr: video.download_addr?.url_list || [],
    };
  });

  await page.goto(TARGET_URL, {
    waitUntil: "domcontentloaded",
    timeout: 30000,
  });

  await page.waitForTimeout(5000);

  console.log(JSON.stringify(result, null, 2));
}

main();
```

## 为什么选 `aweme/detail`

调研过程中发现，`aweme/v1/web/aweme/detail` 是最关键的接口，因为它直接返回整条视频的详情对象。

重点关注字段在：

- `aweme_detail.video.play_addr.url_list`
- `aweme_detail.video.download_addr.url_list`
- `aweme_detail.video.bit_rate`
- `aweme_detail.video.play_addr_h264`
- `aweme_detail.video.play_addr_265`

其中：

- `play_addr` 更接近页面实际播放流
- `download_addr` 更接近平台下载流
- `bit_rate` 适合做清晰度选择

## 从“获取地址”到“触发下载”

在拿到 `downloadAddr` 之后，有几种可能的下载方式：

1. 本地 `curl` 直接下载  
   稳定，但文件会下载到脚本执行机器，而不是远程浏览器机器。

2. 浏览器直接跳转到视频 URL  
   可以工作，但体验一般，容易变成打开一个资源页。

3. 在页面内插入一个临时 `<a download>` 并点击  
   更像浏览器原生下载，且下载发生在远程浏览器机器。

最终采用的是第 3 种方式。

核心代码如下：

```js
const downloadUrl = result.downloadAddr[0] || result.playAddr[0];
const fileName = `${result.awemeId || "douyin-video"}.mp4`;

await page.evaluate(({ url, fileName }) => {
  const a = document.createElement("a");
  a.href = url;
  a.download = fileName;
  a.rel = "noopener";
  a.style.display = "none";
  document.body.appendChild(a);
  a.click();
  a.remove();
}, { url: downloadUrl, fileName });
```

如果配合 CDP 再显式指定下载目录：

```js
const cdp = await context.newCDPSession(page);

await cdp.send("Browser.setDownloadBehavior", {
  behavior: "allow",
  downloadPath: "/tmp",
});
```

那么下载目标就可以明确落到远程浏览器机器的 `/tmp` 目录。

## 调试过程中的关键现象

### 1. 详情接口并不总是稳定返回

在测试过程中，同一条视频链接并不是每次都能稳定拿到 `aweme/detail` 的完整 JSON。

出现过两种情况：

- 正常时：接口返回完整 JSON，可提取 `playAddr` / `downloadAddr`
- 异常时：接口虽然返回 `200`，但响应体为空，导致页面显示“你要观看的视频不存在”

### 2. 清理 cookie / 站点数据后恢复正常

一个非常关键的观察是：

- 浏览器里反复打开同一条短链后，有时会进入异常状态
- 清掉抖音相关 cookie、localStorage、会话数据后，又能恢复正常
- 重启浏览器后，也会恢复正常

这说明问题更像：

- 抖音的站点级风控态
- 安全令牌或本地缓存态被污染

而不是视频本身真的不存在。

### 3. 纯短链跳转本身没有坏

诊断结果表明：

- 短链 `v.douyin.com/...` 仍然能正常 302
- 中间的分享链接也能正常跳转
- 最终仍然能到 `www.douyin.com/video/<aweme_id>`

所以问题不在短链失效，而在于：

- 页面进入目标页之后
- 核心详情接口没有返回有效视频数据

## 当前结论

本次调研已经验证了以下几点：

1. 通过 `playwright-core + connectOverCDP()` 可以控制远程 Chrome。
2. 可以通过抖音详情接口 `aweme/v1/web/aweme/detail` 获取视频真实地址。
3. 可以在远程浏览器内模拟下载，使文件落到远程浏览器机器。
4. 下载目录可以通过 `Browser.setDownloadBehavior` 显式指定。
5. 抖音页面存在一定的会话态/风控态问题，反复访问后可能导致详情接口返回空体。

## 当前最实用的脚本分工

本次调研最终形成了两类脚本：

- `extract-douyin-video.js`
  - 只负责提取视频地址
  - 输出 `playAddr` 和 `downloadAddr`

- `browser-download-douyin.js`
  - 先提取下载地址
  - 再在远程浏览器内触发下载

## 后续可继续优化的方向

如果后续要把这套能力工程化，还可以继续做这些增强：

1. 自动获取最新 `webSocketDebuggerUrl`
   目前 Chrome 重启后 websocket 会变化，最好每次运行时自动从 `/json/version` 获取。

2. 自动选择更高清晰度版本
   当前脚本主要取默认 `downloadAddr`，后续可结合 `bit_rate` 做清晰度选择。

3. 自动检测抖音会话是否异常
   如果页面正文出现“视频不存在”或详情接口返回空体，可自动提示用户重置站点数据。

4. 只清理抖音站点状态，而不是清整个浏览器
   这样可以更精确地恢复可用状态。

---

本次调研最重要的收获不是“写出了一个下载脚本”，而是明确了抖音视频链接提取的关键路径：  
不是从页面表面结构硬猜资源，而是抓住 `aweme/v1/web/aweme/detail` 这个核心接口，从接口结构中提取真实视频地址，再决定后续是本地下载还是浏览器内下载。
