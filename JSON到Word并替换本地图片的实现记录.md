# JSON 到 Word 文档的实现记录

## 目标

把一个 `article.json` 中的文章数据，整理为最终可交付的 `Word` 文档，并满足以下要求：

- 从 JSON 中提取标题、正文、封面图、正文图片
- 确认远程图片路径与本地图片路径一一对应
- 确认本地图片文件真实存在
- 将英文文章翻译为逐句中英对照的 Markdown
- 将 Markdown 内容导出为 `docx`
- 确保 `Word` 文档中显示的是真正的图片，而不是图片路径文本

这份记录的目的，不是只复盘结果，而是沉淀一套以后可以重复使用的实现路径。

## 输入数据结构

本次使用的源文件是当前目录下的 `article.json`。

核心结构如下：

```json
{
  "post": {
    "type": "article",
    "title": "...",
    "text": "...",
    "cover": "远程封面图 URL",
    "cover_path": "本地封面图路径",
    "images": [
      "远程图片 URL 1",
      "远程图片 URL 2"
    ],
    "images_path": [
      "本地图片路径 1",
      "本地图片路径 2"
    ]
  }
}
```

需要重点关注的字段：

- `post.title`
- `post.text`
- `post.cover`
- `post.cover_path`
- `post.images`
- `post.images_path`

## 这次实际完成的流程

## 1. 确认源文件存在

先确认当前目录下确实存在 `article.json`。

可复用思路：

- 优先确认文件存在，再做解析
- 允许当前目录中有其他无权限访问的临时目录，不影响目标文件判断

示例命令：

```bash
rg --files /private/tmp | rg '(^|/)article\.json$'
```

## 2. 检查 JSON 中是否包含所需字段

读取 `article.json`，确认文章核心字段是否齐全。

本次确认结果：

- 有标题
- 有正文
- 有封面图远程路径
- 有封面图本地路径
- 有正文图片远程路径数组
- 有正文图片本地路径数组

示例命令：

```bash
jq '.post | {cover, cover_path, images, images_path}' /private/tmp/article.json
```

## 3. 确认远程图片与本地图片一一对应

这个步骤非常关键，因为后续替换图片时，必须保证远程 URL 和本地文件是严格按顺序对应的。

检查方法：

- 先比较 `images` 和 `images_path` 的长度是否一致
- 再逐项核对数组索引位置是否匹配
- 同时检查封面图的 `cover` 和 `cover_path`

本次结果：

- `images` 数量为 5
- `images_path` 数量为 5
- 文件名按索引完全对应
- 封面图也有对应的本地路径

示例命令：

```bash
jq '.post | {cover, cover_path, image_count: (.images|length), path_count: (.images_path|length)}' /private/tmp/article.json
```

## 4. 检查本地图片是否真的存在

仅仅在 JSON 里有路径还不够，必须确认本地文件实际存在。

这一步的本质是避免后续生成文档时出现：

- 图片插入失败
- 路径有效但文件已被删除
- 部分图片存在、部分图片丢失

本次结果：

- 封面图存在
- 5 张正文图片全部存在

示例命令：

```bash
jq -r '.post | [.cover_path] + .images_path | .[]' /private/tmp/article.json | while IFS= read -r p; do
  if [ -e "$p" ]; then
    printf 'EXISTS\t%s\n' "$p"
  else
    printf 'MISSING\t%s\n' "$p"
  fi
done
```

## 5. 翻译并生成 Markdown 中间稿

在确认文章内容和图片路径无误后，下一步是把文章整理为可输出的 Markdown。

这一步的目标不是直接生成 Word，而是先生成一个稳定的中间产物 `article.md`，方便后续：

- 校对翻译内容
- 检查格式
- 检查图片替换是否符合预期
- 为多种导出格式保留统一来源

本次 Markdown 的要求是：

- 标题包含中文标题
- 按“英文原句 + 中文翻译”逐句对齐
- 只输出最终意译结果，不保留直译和分析过程
- 将正文中的远程图片 URL 替换为对应的本地图片路径

这一步完成后，得到了 `article.md`。

## 6. 第一版 Word 导出为什么失败

最开始使用的是一种“纯文本转 `docx`”的导出方式：

```bash
textutil -convert docx -format txt -inputencoding UTF-8 -output /private/tmp/article.docx /private/tmp/article.md
```

这个方法可以快速生成 `docx`，但有一个根本问题：

- `article.md` 里的图片只是本地路径字符串
- 纯文本转换器不会把路径识别成真正的图片对象
- 结果就是：Word 文档里只有图片路径文本，没有真正的图片显示

这说明：

- “文本里有图片路径” 不等于 “Word 中嵌入了图片”
- 如果目标是可视化图片，必须走真正的图片嵌入流程

## 7. 解决方向判断

出现图片不显示之后，需要先判断问题到底出在哪：

- 是图片文件不存在？
- 是本地路径不正确？
- 是 JSON 数据有问题？
- 还是文档导出方式不支持嵌图？

本次验证后确认：

- JSON 没问题
- 本地图片也都存在
- 路径替换也已经完成
- 真正的问题是导出链路错误：只是文本转换，不是图片嵌入

这一步很重要，因为它决定了后续不是继续修补 Markdown，而是直接更换 Word 生成方式。

## 8. 为什么没有继续依赖 `textutil`

后面尝试验证 `textutil` 能否通过 HTML 方式嵌图，但在当前环境下出现了系统 helper 相关问题，稳定性不够。

这给出的结论是：

- `textutil` 适合做简单格式转换
- 但不适合在当前这套流程中充当“稳定的富文档嵌图引擎”
- 如果要保证结果可靠，最好直接生成真正的 `docx` 结构

## 9. 最终可行方案：直接生成带媒体资源的 `docx`

最终采用的方案，是直接构造一个标准的 `docx` 文件。

本质上，`docx` 是一个 ZIP 包，内部包含：

- 文档 XML
- 关系文件
- 元数据
- 媒体资源

核心思路：

1. 读取 `article.json`
2. 读取已经翻译好的 `article.md`
3. 识别其中哪些行是本地图片路径
4. 将这些图片文件复制进 `docx` 的 `word/media/` 目录
5. 在 `word/document.xml` 中插入真正的图片节点
6. 在 `word/_rels/document.xml.rels` 中建立图片关系
7. 生成最终 `article.docx`

这样做之后，图片就不再是路径文本，而是真正嵌入在 Word 文档里。

## 10. 最终文档的组织方式

本次最终 `docx` 的结构大致如下：

- 文档主标题使用中文标题
- 第二行保留英文原标题
- 封面图作为真正的图片插入文档头部
- 正文保留“英文原句 + 中文翻译”的顺序
- 遇到本地图片路径时，不输出路径文字，而是插入对应图片

最后验证时，文档内成功嵌入了：

- 1 张封面图
- 5 张正文图片

总共 6 张图片

## 关键技术点

## 1. 路径替换只是中间过程，不是最终嵌图方案

很多时候会误以为：

- 只要把远程 URL 替换成本地路径
- Word 就会自动显示图片

实际上不是这样。

路径替换的作用只是：

- 建立“应该使用哪张本地图”的映射
- 为后续真正插图提供依据

真正显示图片，还需要把图片资源嵌入到 `docx` 内部。

## 2. 中间稿 Markdown 很有价值

先生成 `article.md` 是个很好的中间层设计。

原因：

- 便于校对翻译文本
- 便于检查图片替换是否正确
- 后续如果要导出 PDF、HTML、公众号稿、纯中文稿，都可以从这份中间稿继续加工

以后类似流程建议继续保留这一步，不要直接从 JSON 一步跳到 Word。

## 3. 图片存在性检查必须前置

如果不先检查本地图片是否存在，后续会出现两类难定位问题：

- 文档能生成，但部分图片缺失
- 你以为是插图逻辑错了，其实只是源文件不存在

所以这一步最好作为固定前置校验。

## 4. 图片数量与顺序校验必须前置

如果远程 URL 和本地路径不是一一对应，就会导致：

- 封面图错配
- 正文插图顺序错乱
- 文档内容与图片语义不匹配

因此标准检查顺序应该是：

1. 校验数组长度
2. 校验索引顺序
3. 校验本地文件存在

## 5. `docx` 的本质是结构化包，不是单纯文本文件

只要理解了这一点，后面很多问题就会变得清晰：

- 文本内容写在 XML 里
- 图片必须作为媒体资源打包进去
- 文档中显示图片，依赖 XML 节点和关系文件共同生效

这也是为什么“纯文本转 docx”无法自动解决图片显示问题。

## 这套流程以后怎么复用

以后如果再遇到类似任务，可以直接按下面的顺序执行：

1. 确认 JSON 文件存在
2. 提取并检查文章核心字段
3. 校验远程图片与本地图片是否一一对应
4. 校验所有本地图片是否真实存在
5. 先生成 Markdown 中间稿
6. 将中间稿作为内容源，构建最终 `docx`
7. 不要把图片路径当作最终图片方案，必须真正嵌入图片资源
8. 最后校验 `docx` 中是否包含 `word/media/` 资源

## 推荐沉淀为固定脚本的模块

如果后续要把这项工作长期复用，建议拆成 4 个模块：

### 1. JSON 校验模块

输入：

- `article.json`

输出：

- 字段是否齐全
- 图片数量是否一致
- 路径是否存在

### 2. 文本整理模块

输入：

- JSON 标题
- JSON 正文
- 图片映射关系

输出：

- `article.md`

### 3. 翻译模块

输入：

- 英文正文

输出：

- 逐句对照的中英稿

### 4. Word 生成模块

输入：

- 标题
- Markdown 正文
- 本地图片

输出：

- 带内嵌图片的 `docx`

## 本次结论

这次调研最后沉淀出的核心结论是：

- JSON 中的远程图片 URL 可以作为语义来源
- 真正交付时应依赖本地图片路径
- 远程路径替换为本地路径只是准备步骤，不是最终成图步骤
- 如果要在 Word 中显示图片，必须真正把图片嵌入 `docx`
- 先生成 Markdown 中间稿，再导出最终格式，是更稳妥、可复用的流程
