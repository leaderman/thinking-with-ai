# 从让 AI 写代码，到让 AI 直接做事

我的需求很简单：把 X 上的文章保存成本地的 Markdown 文件，图片也一起下载下来。

我的工具也很简单：Chrome CDP（远程调试协议），Chrome 开着的时候，可以向它发送指令，让它帮你做任何事情：导航、点击、执行 JS、截图、下载文件。

我和 AI 一起写了一页操作说明，描述整个流程：

1. 连接 CDP，找到目标页面
2. 等页面加载完
3. 滚动页面触发懒加载图片
4. 提取文字、图片、元数据
5. 下载图片
6. 生成 Markdown

手册里还有一段 Node.js 脚本，把上面六步全部用代码实现。

看起来没什么问题。

## 第一轮：规则写错了

跑起来之后，发现标题不对。

生成的 Markdown 标题显示的是 `Jah Jiren`——作者名，不是文章标题。

原来脚本从浏览器标签页的 `og:title` 里取标题，`og:title` 的格式是：

```
(1) Jah Jiren on X: "I studied 63,000 apps making $10k-100k/mo+" / X
```

脚本用 `.split(' on X')[0]` 截取，结果截到了作者名而不是文章标题。

AI 修了正则，改成从引号里提取——好了，这次对了。

但很快又发现：另一篇文章根本没有 `og:title` 能解析的格式，因为那篇文章根本没有 `<h1>` 标签，第一行文字也是作者名。

于是 AI 又加了一层逻辑：先找 `<h1>`，再解析 `og:title`，再降级到第一行文字。

改了，能用了。

## 第二轮：噪声永远清不干净

X 的页面 `innerText` 里混了大量 UI 文字：点赞数、转发数、「Reply」、「Repost」、时间戳、「Want to publish your own Article?」……

于是我们加了一批过滤规则：

```javascript
if (/^[\d\s.,KMB%]+$/.test(t)) return false;  // 纯数字行
if (/^\d{1,2}:\d{2}\s+(AM|PM)/i.test(t)) return false;  // 时间戳
if (/^(Reply|Repost|Want to publish|Upgrade to Premium...)/i.test(t)) return false;
```

规则加了十几条，每次跑完发现还有漏网之鱼，就再加一条。

这时候我开始觉得不对劲——**这条路走不通**。

X 随时会更新 UI，新版本出现新的噪声文字，规则就又失效了。我们永远在追赶，永远追不上。

## 关键转折：一个根本性的问题

思考了一会儿，我给 AI 提了一个问题：

> "我觉得现在使用代码来提取文章是不是本身就是有问题？因为代码的规则本身就是死的，我们不可能把所有的场景全部都考虑到。"

这句话给 AI 和我自己打开了一扇门。

代码的本质是规则，规则是对已知场景的总结。但现实总有新场景。用代码来处理「什么是噪声、什么是正文、什么是标题」，本质上是在用刚性工具解决柔性问题。

我们重新想了一下分工：

- **代码**：只处理规则固定的机械操作——连接浏览器、导航、滚动、下载图片、写文件
- **AI**：处理所有需要语义理解的工作——识别标题、清理噪声、判断格式、生成 Markdown

脚本因此大幅简化。它不再做任何理解工作，只是把页面原始内容「搬」下来，交给 AI。

这是整个过程里最重要的一次转向。

## 第三轮问题：信息被悄悄丢掉了

新方案跑起来之后，Markdown 的内容质量明显提升——噪声少了，标题对了，正文流畅了。

但又发现了新问题：

有一篇文章里，原文有**有序列表**（1. 2. 3.），保存后变成了**无序列表**（-）。原文里用加粗段落作为强调，保存后变成了 bullet point，而且把粗体陈述和后面的解释说明挤在了同一行。

为什么会这样？

因为现在传给 AI 的是 `innerText`。

`innerText` 会把所有 HTML 格式信息全部抹掉：`<ol>` 变成文字，`<strong>` 变成文字，`<blockquote>` 变成文字。AI 拿到的是一份纯文字，只能靠语义去猜原来的格式是什么。猜，就会猜错。

数据管道是这样的：

```
HTML DOM → innerText（格式信息全丢）→ AI 猜格式 → Markdown
```

信息在第一步就丢了，后面再聪明也补不回来。

## 弯路：放弃 innerText，改为解析 HTML

发现格式丢失之后，AI 的第一反应是换一种提取方式。

`innerText` 的问题很清楚：它把所有 HTML 结构抹掉，只留纯文字。

那就不用 `innerText`，改为直接读 HTML，再写一个 DOM walker 遍历节点——遇到 `<ol>` 输出有序列表，遇到 `<strong>` 加粗，遇到 `<pre>` 变代码块，遇到 `<blockquote>` 加引用符号。每种格式对应一条处理逻辑。

这一次 AI 还没有动手就被我拦住了。

X 的 HTML 结构复杂，格式和样式无穷无尽，永远枚举不完的，不能再走规则的老路了。

## 真正的解法：不要丢信息

有了之前的经验，这里我悟了。

**问题不是穷举规则，是信息不能丢。**

HTML 里本来就有完整的格式信息。`<ol>` 是有序列表，`<strong>` 是粗体，`<blockquote>` 是引用块。我们不需要重建它，只需要不丢弃它。

脚本的输出由一份变成了三份：

- **`raw.json`**：基于 `innerText` 的文字序列 + 图片位置（用于读正文内容）
- **`article-clean.html`**：清洗后的语义 HTML，去掉所有 `class`/`style`/`data-*` 属性，只保留 `<ol>`、`<strong>`、`<blockquote>` 这类语义标签（用于理解格式结构）
- **`article-raw.html`**：原始 HTML，未做任何处理（终极参考）

更狠的是，我还加上了页面截图。

AI 现在有四份参考资料可以按需取用：

| 数据 | 用途 |
|------|------|
| `raw.json` | 读正文内容、确定图片位置 |
| `article-clean.html` | 判断格式结构 |
| `article-raw.html` | 遇到歧义时的终极参考 |
| `screenshots/` | 视觉验证和格式校验 |

AI 不再需要猜格式。但更重要的是，AI 现在可以自己决定看哪份数据。

大多数时候，`raw.json` 里的文字序列就够用了——读正文、确认图片位置，不需要关心格式。
需要判断某段内容是有序列表还是无序列表、是粗体强调还是普通文字，打开 `article-clean.html` 看一眼，`<ol>` 就是有序，`<strong>` 就是粗体，清清楚楚。
遇到还拿不准的边界情况，`article-raw.html` 是最后的参考，原始 HTML 完整保留，没有任何处理。
再不行，文字描述不清楚的时候，还有截图，按滚动顺序保存的页面截图直接对照视觉来确认。

## 效果：出乎意料地好

格式对了，噪声没有了。

再也不需要反复调规则了。

把页面的原始数据收集下来，然后交出去。
剩下的，标题是什么、哪段是正文、格式结构怎么理解、Markdown 怎么生成，全部由 AI 自己判断、自己处理。
没有人告诉它"这行是噪声"，没有人告诉它"这里应该用有序列表"，它自己看、自己做决定。

## 回头看：这个过程在说什么

复盘这整个过程，有几件事值得记下来。

**第一，代码不适合处理柔性问题。**

代码擅长「如果是这个，就做那个」。但「这段文字是噪声还是正文」这种判断，没有明确边界，边界会随着平台更新而变动。把这类判断交给代码，只是在把复杂性转移到规则集里，规则集会越来越臃肿，越来越脆。

**第二，分工的边界应该在「理解」这里划。**

不需要理解的，交给代码——规则固定，稳定可靠。需要理解的，交给 AI——语义灵活，适应变化。这条边界一旦划清，两边各自做擅长的事，整个系统就稳了。

**第三，信息丢失是最隐蔽的错误。**

`innerText` 拿到手感觉很干净，但格式信息已经不在了。这种损失不报错、不崩溃，只是在后续步骤里悄悄变成质量问题。发现它需要多走几步——先用，发现问题，追溯根因，才能看到那个悄悄发生的丢失。

保留冗余信息（多输出几份格式），让后续步骤有足够的上下文来工作，是比「用最少的数据」更可靠的策略。

**第四，AI 的角色变了。**

最开始，AI 是一个工具——我描述需求，它帮我写代码，写完交给我，它的工作就结束了。真正干活的是代码，代码是死的，它只能按规则走。

后来，AI 变成了队友。每一次转向都是在对话里发生的。「这条路走不通」不是我一个人想出来的，也不是 AI 告诉我的，是两边来回说了几句之后自然收敛到的。
保留哪些数据、分工怎么划、截图有什么用——这些决策没有一个是一方提出、另一方执行的。

再后来，AI 又往前走了一步——变成了执行者。
不是执行代码，是执行意图。
我只需要写清楚「要做什么」、「要什么结果」，「怎么做」留给它自己判断。
操作说明不需要把每个细节都覆盖到，AI 读懂方向，自己去理解、自己去决定。

从工具，到队友，到智能的执行者，新的世界打开了。
